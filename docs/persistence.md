# Persistence

This document defines Lobster persistence behavior for both local and daemon execution modes.

## Goals

- Use SQLite as the primary embedded datastore.
- Use sqlc for type-safe query generation.
- Keep deterministic behavior across local and remote execution.
- Preserve file report outputs for CI compatibility while introducing structured run history.

## Storage engine

- Database: SQLite
- Query layer: sqlc-generated Go code
- Migration model: explicit, versioned SQL migrations

## Company standard alignment

Lobster persistence follows the same BCP codegen-first conventions used in other backend services:

- SQL is authored by developers; repositories are generated by sqlc.
- Schema and queries are source-of-truth artifacts reviewed in PR.
- Generated data-access code is committed but never hand-edited.
- Migration discipline and compatibility checks are mandatory for every schema change.

## Scope of persisted data

The following data is persisted in SQLite:

- run metadata and run history
- feature/scenario/step result records
- plan metadata and plan artifact blob references
- minimal adapter and stack operational metadata required for deterministic lifecycle handling

Default blob backend rule:

- plan artifact blobs are stored on local filesystem under `.lobster/plans` by default.
- SQLite stores metadata and blob references, not the full blob payload.

The following outputs remain file-based for compatibility:

- JSON report files
- JUnit XML files

Authority rule:

- SQLite-backed API state is authoritative for completed runs and run history.
- JSON/JUnit artifacts are export formats derived from authoritative state.

## Access model by execution mode

Local mode:

- CLI process may read/write SQLite directly.

Daemon mode:

- Daemon is the single writer for SQLite.
- External clients (CLI, Wish, HTTP) must access state through daemon API only.
- Direct external writes to the SQLite file are not supported.

This single-writer rule avoids cross-process lock contention and undefined state behavior.

## Shared DB topology

Lobster supports a shared SQLite data model between local and daemon usage, with strict access policy:

- same schema and migration lineage in both modes
- daemon-authoritative writes in daemon mode
- local direct writes only when running in local mode

Monorepo isolation rule:

- use one SQLite database per selected workspace path.

## SQLite runtime settings

Recommended defaults:

- journal mode: WAL
- foreign keys: ON
- busy timeout: configured to reduce transient lock failures
- synchronous mode: NORMAL for local speed, configurable for stricter durability

## sqlc workflow

- SQL files in versioned query packages are authoritative for data access.
- sqlc output is generated during build/dev workflows.
- Application code should call repository abstractions over raw SQL where practical.

Recommended sqlc source layout:

- `sql/run/queries.sql` for run lifecycle state
- `sql/plan/queries.sql` for plan artifacts and metadata
- `sql/stack/queries.sql` for stack/runtime metadata
- split into `queries_<entity>.sql` files when a surface grows

Recommended sqlc output layout:

- `gen/sqlc/run/`
- `gen/sqlc/plan/`
- `gen/sqlc/stack/`

Recommended query naming:

- `Get<Entity>` for single record reads
- `List<Entity>` for collection reads
- `Create<Entity>` for inserts
- `Update<Entity>` for updates
- `Delete<Entity>` for soft delete behavior
- `Count<Entity>` for filtered counts

Generated package shape should include `db.go`, `models.go`, and `queries.sql.go`.

## Schema ownership and change policy

- Lobster core owns schema and migrations.
- Schema changes require migration scripts in the same change set.
- Destructive migration changes require explicit release notes and upgrade guidance.

## Migration policy

Migration execution mode aligns with runtime configuration:

- `auto`: Lobster applies pending migrations on startup.
- `external`: migrations are managed outside Lobster; startup validates expected schema version.
- `disabled`: migration checks are skipped; intended only for controlled scenarios.

Compatibility rules:

- If DB schema is newer than the binary supports, startup fails with a clear version error.
- If DB schema is older and migration mode allows upgrades, pending migrations are applied.

Migration file conventions:

- sequential number prefix with descriptive suffix
- `.up.sql` and `.down.sql` pairs for each change
- idempotent guards where possible (`IF EXISTS`, `IF NOT EXISTS`)

Required policy:

- every migration must include a paired down migration file.

Safety standards:

- use expand-contract for high-risk changes
- separate large data migrations from schema migrations
- avoid long-running write locks in startup migration paths

Example naming:

- `migrations/000001_initial_schema.up.sql`
- `migrations/000001_initial_schema.down.sql`
- `migrations/000002_add_run_indexes.up.sql`
- `migrations/000002_add_run_indexes.down.sql`

## Retention policy

Default retention behavior:

- keep 500 runs and 90 days of run metadata and detailed step data by default
- allow configurable retention windows by age and count
- support explicit cleanup command behavior in a future CLI or admin maintenance surface, once that command contract is defined

Event stream retention default:

- retain async run event streams for 30 days by default.

Retention settings should never remove active run data.

## Locking and concurrency

Current baseline:

- v0.1 execution remains deterministic and serial at scenario level.
- daemon may still serve multiple client requests over time; DB writes are serialized by server logic.

Future parallelism:

- schema and query patterns must be reviewed before enabling higher write concurrency.
- the v0.2 concurrency plan is server-side write serialization with explicit queueing, lock-contention metrics, and no parallel scenario execution yet.
- any later move beyond serial scenario execution needs an explicit lock-contention strategy, write-queue policy, and retention-job interaction model.
- long-running write transactions should be minimized.

Scope note:

- parallel scenario execution guarantees are explicitly out of scope for the initial persistence contract.

Operational guidance for SQLite:

- keep write transactions short and explicit
- prefer server-side queueing over client-side write retries
- expose lock contention metrics in daemon health diagnostics

## Security and secrets

- Sensitive values must not be stored in plaintext unless explicitly required.
- Redaction policy applies to logs and exported diagnostics.
- Credentials and tokens should stay in environment/secret stores, not in durable run records.

## Operational visibility

Lobster should provide:

- schema version and migration status in config/health outputs
- clear lock and migration errors
- diagnostics for DB path, mode, and retention policy

## Non-goals

- This document does not define public RPC shapes.

See [api-reference.md](api-reference.md) for protocol and transport contracts.
